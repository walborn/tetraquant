# Быстрый старт: Создание нового проекта

Этот репозиторий является **шаблоном (Boilerplate)** для создания новых фронтенд-сервисов. Но лучше воспользоваться [start from scratch](./start-from-scratch.md)

Следуйте этой инструкции, чтобы развернуть новое приложение.

## 1. Инициализация проекта

### Вариант А: Использование GitHub Template
Если репозиторий помечен как Template, нажмите кнопку **"Use this template"** в интерфейсе GitHub.

### Вариант Б: Кронирование вручную
Скопируйте репозиторий и очистите историю git, чтобы начать с чистого листа..

```bash
# 1. Клонируем шаблон в новую папку (замените my-new-service на имя вашего проекта)
git clone <URL_THIS_REPO> my-new-service

# 2. Переходим в папку
cd my-new-service

# 3. Удаляем привязку к исходному репозиторию
rm -rf .git
git init
git branch -M master
```

## 2. Установка зависимостей

Мы используем **Bun** как пакетный менеджер.

```bash
bun install
```

## 3. Первичная настройка

### Конфигурация
1.  Откройте `package.json` и обновите поле `name` на имя вашего сервиса.
2.  Создайте файл переменных окружения:
    ```bash
    cp .env.example .env
    ```
    *(Если `.env.example` отсутствует, создайте `.env` вручную)*

    Пример `.env`:
    ```env
    VITE_BACKEND=http://localhost:8080
    ```
    > `VITE_BACKEND` — адрес вашего бэкенд-сервиса. Все запросы на `/api`, `/sso` и `/rpc` будут проксироваться туда (см. `vite.config.ts`).

### Запуск в режиме разработки
```bash
bun dev
```
Приложение будет доступно по адресу `http://localhost:3000`.

## 4. Авторизация (Yandex SSO)

В наших сервисах используется корпоративная авторизация через **Yandex SSO**.
В шаблоне уже настроено проксирование запросов, связанных с аутентификацией.

**Как это работает:**
1.  frontend (`vite.config.ts`) проксирует все запросы, начинающиеся с `/sso`, на бэкенд (`VITE_BACKEND`).
2.  Бэкенд берет на себя OAuth flow (redirect на Яндекс, валидация коллбэка, выдача Session Cookie).
3.  Фронтенд не хранит секретов (Client Secret).
4.  Для проверки авторизации обычно используется эндпоинт `/api/user/me` (или аналогичный), который возвращает профиль текущего пользователя или 401.

**Ваша задача как разработчика:**
- Убедиться, что `VITE_BACKEND` указывает на рабочий бэкенд (локальный или стейдж).
- Обрабатывать 401 ошибки (редирект на `/sso/login` или показ модального окна).

## 5. Полезные команды

*   `bun dev` — запуск dev-сервера.
*   `bun run build` — сборка продакшен билда (Nitro + Vite).
*   `bun run serve` — предпросмотр собранного приложения.
*   `bun run lint` — проверка линтером (Biome).
*   `bun run test` — запуск тестов (Vitest).
